# ขายดี Stock — Index Redesign Design

**วันที่:** 2026-07-13
**สถานะ:** อนุมัติทิศทางดีไซน์แล้ว รอผู้ใช้ตรวจสเปกก่อนวางแผนลงมือ

## เป้าหมาย

ออกแบบหน้า index ของ “ขายดี Stock” ใหม่ทั้งหมดให้เป็นเว็บไซต์ขายระบบ POS และคลังสินค้าสำหรับร้านค้าปลีกสมัยใหม่ เช่น มินิมาร์ท ร้านโชห่วย ร้านขายส่ง ร้านเครื่องสำอาง และร้านอะไหล่ หน้าเว็บต้องดูมืออาชีพ เข้าใจง่าย และสื่อทันทีว่าเป็นระบบสำหรับร้านขายสินค้า ไม่ใช่ร้านอาหารหรือคาเฟ่

เป้าหมายหลักของหน้า คือพาผู้เยี่ยมชมจากการเข้าใจคุณค่าของระบบ ไปสู่การกดสมัครใช้งานฟรีหรือดูตัวอย่างระบบจริง

## ทิศทางภาพ

ใช้แนวทาง **Retail Command Center**:

- สีหลักเป็นน้ำเงินเข้ม สีเสริมเป็นมิ้นต์และฟ้าสว่าง บนพื้นขาวอมเทา
- ภาพลักษณ์ทันสมัย น่าเชื่อถือ และเข้าถึงได้สำหรับร้านค้ารุ่นใหม่
- ใช้หัวข้อทรงเรขาคณิตที่รองรับภาษาไทย และเนื้อหาที่อ่านง่าย
- ใช้การ์ดมุมโค้ง ขอบบาง เงานุ่ม และพื้นที่ว่างที่โปร่ง
- หลีกเลี่ยงความเป็นไซไฟจัด แสงนีออนมากเกินไป หรือองค์ประกอบที่ทำให้ดูเหมือนฟินเทค

## ภาพ Hero

ใช้ภาพ **Retail Command Center** ที่สร้างด้วย ImageGen เป็นภาพพื้นหลัง Hero แบบเต็มความกว้าง ภาพแสดงพนักงานร้านค้าปลีกกำลังใช้เครื่อง POS พร้อมชั้นสินค้าด้านหลัง และมีพื้นที่ว่างทางซ้ายสำหรับข้อความ

องค์ประกอบ Hero:

- Navigation แบบโปร่งเล็กน้อย แต่ข้อความและปุ่มต้องมี contrast ชัดเจน
- กล่องข้อความกระจกฝ้าแบบ Glassmorphism ทางซ้าย โดยมีพื้นทึบเพียงพอให้อ่านได้บนทุกส่วนของภาพ
- Headline: “ขายคล่อง สต็อกตรง รู้กำไรทุกวัน”
- ข้อความรองอธิบายว่าเป็น POS + คลังสินค้าสำหรับร้านขายของ
- CTA หลัก “เริ่มใช้ฟรี” และ CTA รอง “ดูระบบจริง”
- การ์ดข้อมูล POS ขนาดเล็ก เช่น ยอดขายวันนี้ บิลล่าสุด และสต็อกใกล้หมด วางทับภาพโดยไม่บังใบหน้าหรือเครื่อง POS
- บนมือถือ กล่องข้อความต้องกินความกว้างเกือบเต็มจอ ภาพครอปโดยรักษาพนักงานและ POS และลดจำนวนการ์ดลอยเพื่อไม่ให้รก

หากภาพโหลดไม่ได้ Hero ต้องยังใช้งานได้ด้วยพื้นหลังไล่สีน้ำเงินเข้มและข้อความครบถ้วน

## Live Retail Ticker

วางแถบ Ticker ต่อเนื่องที่ด้านล่าง Hero เพื่อแสดงข้อมูลตัวอย่างจากระบบ เช่น:

- ยอดขายวันนี้ ฿12,450
- บิลล่าสุด #087
- สต็อกใกล้หมด 4 รายการ
- กำไรวันนี้ +18%
- รับโอนสินค้าเข้าคลัง 12 ชิ้น

Ticker ต้อง:

- เลื่อนต่อเนื่องจากขวาไปซ้ายอย่างนุ่มนวลและไม่เร็วเกินไป
- ใช้พื้นน้ำเงินเข้มทึบ ตัวอักษรขาว และใช้สีมิ้นต์เฉพาะข้อมูลสำคัญ
- แยกรายการด้วยเส้น จุดสถานะ หรือไอคอนเส้นที่อ่านง่าย
- หยุดเมื่อ hover หรือ focus
- ไม่ใช้พื้นเบลอหลังข้อความ
- ใช้ track ซ้ำเพื่อให้ลูปต่อเนื่องโดยไม่มีจังหวะกระโดด
- เมื่อเปิด `prefers-reduced-motion` ให้หยุด animation และแสดงข้อมูลเป็นรายการนิ่งที่ wrap ได้

## โครงสร้างหน้า

ลำดับเนื้อหา:

1. Hero + Live Retail Ticker
2. แถบผลลัพธ์หลัก: ขายเร็วขึ้น, สต็อกตรงทุกบิล, ดูกำไรทันที
3. Product Showcase แสดงหน้าจอ POS จริงในรูปแบบ code-native mockup ประกอบด้วยสินค้า ตะกร้า ยอดรวม และวิธีชำระเงิน
4. Workflow: ตั้งสินค้า → เปิดขาย → ตัดสต็อก → ดูรายงาน
5. Feature Bento: คลังสินค้า, หลายสาขา, สมาชิก, โปรโมชั่น, ลูกหนี้ และสิทธิ์พนักงาน
6. ส่วน “สร้างมาเพื่อร้านขายของ” พร้อมตัวอย่างประเภทร้านที่สอดคล้องกับฟีเจอร์จริง
7. Pricing ที่ดึงข้อมูลแพ็กเกจจริงจากระบบเดิม
8. FAQ ที่ตอบข้อกังวลเรื่องการเริ่มใช้ อุปกรณ์ การชำระเงิน และข้อมูล
9. CTA ปิดท้ายและ Footer แบบกระชับ

ไม่ใช้ตัวเลขจำนวนลูกค้า รีวิว หรือคำกล่าวอ้างที่ตรวจสอบไม่ได้ ความน่าเชื่อถือจะมาจากการแสดงฟีเจอร์และสถานการณ์ใช้งานจริง

## สถาปัตยกรรมและคอมโพเนนต์

- คง `app/page.tsx` เป็น Server Component เพื่อเรียกสถานะล็อกอินและข้อมูลแพ็กเกจได้โดยตรง
- แยกส่วนที่มีขอบเขตชัดเจนเป็นคอมโพเนนต์ใน `components/landing/` เช่น Hero, LiveTicker, ProductShowcase และ FeatureGrid
- เก็บเนื้อหาที่ทำซ้ำเป็น typed data arrays เพื่อแก้ไขข้อความได้จากจุดเดียว
- ใช้ CSS/Tailwind สำหรับ Ticker และ micro-interactions เป็นหลัก ไม่เพิ่ม runtime UI dependency
- ใช้ icon แบบ SVG ในชุดเส้นเดียวกัน ไม่ใช้ emoji เป็นไอคอนโครงสร้าง
- เก็บภาพ ImageGen ที่เลือกไว้ใน `public/images/landing/` และอ้างอิงผ่าน `next/image`
- รักษาการเชื่อมต่อเดิมกับสถานะล็อกอิน, เส้นทาง `/signup`, `/login`, `/dashboard`, `/pricing` และข้อมูลจาก `lib/plans.ts`

## Responsive และ Accessibility

- รองรับอย่างน้อย 375px, 768px และ 1440px โดยไม่มี horizontal scroll
- ขนาดข้อความเนื้อหาบนมือถือไม่น้อยกว่า 16px ในส่วนหลัก
- ปุ่มและลิงก์มีพื้นที่กดอย่างน้อย 44×44px
- ทุก interactive element มี focus ring ที่เห็นชัด
- สีข้อความปกติต้องมี contrast อย่างน้อย 4.5:1
- ใช้ heading hierarchy ตามลำดับ และมี semantic landmarks
- ภาพ Hero เป็นภาพตกแต่ง จึงกำหนด `alt=""`; ความหมายหลักทั้งหมดต้องอยู่ในข้อความ HTML
- Animation ใช้ transform เท่านั้น และรองรับ `prefers-reduced-motion`
- Ticker ต้องไม่เป็นแหล่งข้อมูลสำคัญเพียงจุดเดียว; ข้อมูลเป็นตัวอย่างประกอบและหน้าหลักยังเข้าใจได้เมื่อ animation ถูกปิด

## Performance และ Failure States

- กำหนดขนาดและสัดส่วนภาพล่วงหน้าเพื่อลด layout shift
- Hero image เป็นภาพสำคัญเหนือ fold จึงโหลดแบบ priority และมี responsive sizes ที่เหมาะสม
- ส่วนที่อยู่ใต้ fold ใช้การโหลดภาพแบบ lazy ตามค่าเริ่มต้น
- ลด JavaScript ฝั่ง client โดยใช้ Server Components และ CSS animation
- ถ้าภาพ Hero โหลดล้มเหลว พื้นหลังสีและ gradient ต้องรักษาความอ่านง่ายและ CTA ทั้งหมด
- หลีกเลี่ยง animation ที่บล็อก input หรือทำให้ layout reflow

## การตรวจสอบ

- รัน lint และ production build
- ตรวจ visual layout ที่ 375px, 768px และ 1440px
- ตรวจ Hero crop, ความชัดของ Glassmorphism, Ticker loop และการหยุดเมื่อ hover/focus
- ตรวจ reduced-motion, keyboard navigation และ focus states
- ตรวจ contrast, touch targets และ heading hierarchy
- ตรวจว่าไม่มี horizontal scroll หรือเนื้อหาถูก fixed/sticky element บัง
- ตรวจเส้นทาง CTA ทั้งสถานะล็อกอินและยังไม่ล็อกอิน
- ทำ browser visual QA และแก้จนหน้าไม่มีข้อผิดพลาดด้านเลย์เอาต์ที่มองเห็นได้

## ขอบเขตที่ไม่ทำ

- ไม่เปลี่ยน backend, database schema, authentication หรือ logic ของ POS
- ไม่สร้างตัวเลขลูกค้า รีวิว หรือข้อมูล social proof ที่ไม่มีแหล่งยืนยัน
- ไม่เปลี่ยนชื่อแบรนด์ “ขายดี Stock”
- ไม่เพิ่ม framework หรือ component library ใหม่
